# 产品教练

> 让 ChatGPT 不只是回答一次问题，而是持续理解你正在做的产品，守住已经确认的方向，帮你判断下一步，并把想法推进成真正可实现、可验证、可迭代的产品。

`product-coach-router` 是一个面向真实产品项目的 ChatGPT Skill，中文显示名为“产品教练”。

它不是一份 PRD 模板，也不是一套要求你照着走的标准流程。它更像一个长期参与项目的资深产品搭档：读取会议记录、原型、PRD、代码、截图和用户反馈，先还原“当前产品到底是什么”，再判断这轮最应该解决什么，并直接完成对应工作。

## 它解决什么问题

使用 AI 做产品时，真正困难的往往不是“写不出一份文档”，而是：

- 资料散落在会议记录、聊天、截图、Demo、代码和不同版本文档里；
- 每次开启新对话，AI 都像第一次接触项目，容易重新发散；
- 已经确认的功能会在改文档、改界面或开发时被悄悄改变；
- 产品、设计、工程、AI 和 QA 各说各话，最后没人能说明当前版本到底应该是什么；
- 文档写了很多，但产品仍然无法操作、无法验证，也不知道下一步应该做什么；
- 独立产品经理能够借助 AI 完成很多工作，却缺少一个持续帮助自己判断、收敛和验收的搭档。

产品教练的核心作用，是让一个项目在多轮讨论、多人协作和 AI 开发中仍然保持连续：

1. **看清当前产品**：从已有资料中还原已确认的方向、规则、现状和冲突；
2. **守住产品事实**：区分用户决定、证据、建议、假设和待确认问题，避免 AI 擅自改需求；
3. **判断下一步**：不机械重走完整流程，而是找到当前最早、最重要的未解决问题；
4. **直接完成工作**：输出真正可使用的调研结论、设计原则、PRD、开发任务、验收报告或迭代方案；
5. **验证而不是自我宣布完成**：通过实际文件、构建、运行、操作和检查确认结果；
6. **在做事中提升产品判断**：说明关键取舍，但不把工作变成产品管理课程。

## 核心理念

### 1. 产品开发不是文档流水线，而是一连串需要被持续维护的判断

会议记录、PRD、原型、代码和反馈只是载体。真正需要保持一致的是：产品服务谁、解决什么问题、用户如何使用、哪些规则已经确认、哪些仍然未知。

因此，产品教练首先维护“当前产品事实”，而不是看到新任务就重新生成一套方法论和文档。

### 2. 从当前产品继续，不从标准流程重新开始

用户可能已经有 Demo、代码、旧 PRD、截图和多轮反馈。产品教练会先读懂现状，找出其中已经成立、已经过时、互相冲突和仍未决定的部分，然后从最需要推进的位置继续。

它不会因为发现资料很多，就要求重新做一轮调研、重新写一份 PRD 或重新设计整个产品。

### 3. AI 可以分析和建议，但不能替用户做未经授权的产品决定

产品教练会主动发现问题、比较方案并给出推荐，但会明确区分：

- 已确认事实；
- 有来源的事实；
- AI 推断；
- AI 建议；
- 必须由用户决定的问题。

未经确认的新角色、新审批、新权限、新商业规则和重大交互，不会被偷偷写成正式产品事实。

### 4. 文档的价值是让产品更清楚、更容易实现和验收

产品教练不是为了“文档看起来专业”而堆术语。正式 PRD 应让读者理解整个产品如何运行；开发任务应让 Codex 或工程师能够直接实现；QA 应检查真实行为，而不是检查开发者写的完成说明。

任何文档都只是推进产品的工具，不是最终目标。

### 5. 产品必须通过真实使用被验证

能打开页面、能看到卡片、按钮能点击，不代表产品已经成立。关键任务需要能够完整操作，用户需要看到加载、失败、恢复、确认和结果去向，多个角色和页面之间的变化也需要真实发生。

这些内部质量标准由 Skill 负责，使用者不需要学习一套等级术语。

### 6. 保留已经做对的东西，只修改真正需要改变的部分

产品迭代不是每轮推倒重来。已经确认的设计、有效代码、成熟交互和高质量文档都应被保护。新反馈只修改它真正影响的层级：设计理念、PRD、工程规则、模拟数据、实现或缺陷。

## 它可以帮你完成什么

### 从混乱材料中找到真实问题和方向

输入客户会议、访谈、老板意见、竞品资料或销售转述，产品教练会区分事实、观点、诉求、方案、决定和待确认问题，判断信息是否足够进入产品设计，并给出下一步最小行动。

### 把长期判断沉淀为稳定的产品设计原则

当多轮反馈形成稳定规律，或新的意见与既有方向冲突时，产品教练会提炼候选原则、说明适用范围和取舍，并在用户确认后维护。它不会把一次临时意见随便升级为长期理念。

### 生成真正能指导设计、开发和验收的正式 PRD

它会基于全部权威资料生成一份唯一的当前产品说明，覆盖产品结构、页面关系、用户流程、内部运作、详细功能、状态、异常、恢复、数据去向、跨端影响和验收标准。

它不会为了格式整齐而压缩复杂功能，也不会把 API、数据库和开发过程塞满主 PRD。

### 把产品设计推进成可以实际操作和验证的版本

基于 PRD、原型和现有代码，产品教练可以生成 Codex 可执行的完整开发包，让产品经理能够真正走完主要任务、体验状态变化、制造失败场景、检查跨页面和跨角色结果，而不是只看静态 Demo。

### 审查实现，区分产品问题和代码问题

它会实际检查已有项目、页面、构建和关键流程，区分：

- 产品设计本身不合理；
- 实现偏离了已确认设计；
- 交互或视觉体验不足；
- 状态、异常和恢复没有完成；
- 当前版本距离客户试点或正式上线还缺什么。

### 根据真实反馈持续迭代

输入用户操作反馈、客户意见、截图、数据或 QA 结果，产品教练会判断应该修改设计理念、PRD、工程规则、模拟数据还是当前实现，并保护不受影响的部分。

## 怎么使用

### 安装后，不需要自己选择模块

你不需要判断当前属于“调研模块”“PRD 模块”还是“QA 模块”。直接描述你正在做的事，产品教练会读取资料、判断当前阶段并选择合适的工作方式。

可以在消息开头写“调用产品教练”，也可以直接提出任务。

### 第一次处理一个项目

把能够说明项目现状的资料放进同一个 ChatGPT 项目或当前对话，例如：

- 会议记录和客户反馈；
- 现有 PRD、功能设计文档和设计原则；
- 截图、原型或 Demo；
- 当前代码或项目压缩包；
- 你本轮最新确认的意见。

然后直接说：

```text
调用产品教练。先读完这些资料，告诉我当前产品已经确定了什么、哪些地方互相冲突、现在最应该先解决什么，然后直接完成这一步。
```

### 持续推进一个长期项目

在项目工作区中保存当前产品事实、设计原则、重要决定、待确认问题和当前交付物。之后每轮只需要提供新增资料和本轮目标。

产品教练会从当前状态继续，而不是重新理解整个项目。任何重要事实只有在用户确认后才会正式更新。

### 它每轮应该怎样工作

正常情况下，产品教练会：

1. 先读取已有资料，不问资料中已经有答案的问题；
2. 用简短语言说明当前判断和本轮真正要解决的问题；
3. 指出来源冲突、信息缺口和需要用户决定的事项；
4. 直接生成、修改或审查本轮需要的真实产物；
5. 说明完成证据、未解决问题和下一步，而不是只给一套建议。

只有确实无法通过已有资料判断的重大产品问题，才需要用户做决定。

## 常用示例

### 整理客户会议

```text
这是一次客户会议记录。请区分客户事实、客户观点、老板诉求、真实用户问题、已经形成的决定和仍需确认的问题，并判断我们是否应该进入产品设计。
```

### 继续已有产品，而不是重新设计

```text
这是当前 PRD、最新截图和项目代码。先还原当前产品真实状态，找出文档和实现之间的冲突，不要从头设计，也不要改动已经确认的功能。
```

### 生成正式 PRD

```text
基于我指定的全部权威资料，生成唯一正式产品功能 PRD。信息不足时明确写待确认，不得自行补充新角色、新流程或新权限。
```

### 让 Codex 完整实现并供产品经理验证

```text
基于当前 PRD 和代码，为 Codex 生成完整的产品设计验证开发任务。我要能够实际操作所有主要功能、异常和跨端变化，而不是得到一套静态页面。
```

### 验收 Codex 的结果

```text
实际检查当前项目，不要相信完成说明。告诉我哪些功能真的完成了，哪些偏离了设计，哪些只是看起来能用，并生成下一轮修复任务。
```

### 处理新反馈

```text
这是试用反馈和截图。判断每条问题属于产品方向、设计原则、PRD、工程规则、实现缺陷还是体验问题，并只修改真正受影响的部分。
```

## 它与常见工具有什么不同

### 它不是普通 PRD 生成器

PRD 只是产品生命周期中的一个产物。产品教练会先确定当前产品事实，并负责把 PRD 与后续实现、验收和迭代连接起来。

### 它不是产品管理课程

它会在关键节点解释判断方法，但主要任务始终是完成真实工作，而不是要求用户先学习完整方法论。

### 它不是提示词合集

用户不需要记住一组复杂提示词，也不需要手工选择多个 Agent。一个入口会根据当前任务加载对应专业规则。

### 它不是自动替用户做决定的项目经理

AI 可以发现问题、提供方案和执行大量工作，但产品方向、重要取舍和高影响决定仍由用户负责。

### 它不是要求所有项目走同一套流程

简单、可逆的修改可以快速完成；复杂、跨角色、涉及 AI 或客户数据的产品会使用更严格的规格和验证。严谨度取决于风险，而不是为了流程完整而增加工作。

## 适合谁

- 独立产品经理或小团队产品负责人；
- 正在使用 ChatGPT、Codex 等工具推进真实产品的人；
- 已经有多轮资料、多个版本和现有代码，需要继续而不是重做的项目；
- 希望 AI 不只写文档，还能帮助判断、开发、验收和迭代的人；
- 正在学习产品工作，但不想先读完一套产品管理课程的人。

## 安装

1. 打开仓库的 **Actions**；
2. 进入 **Build Product Coach Skill** 的最新成功运行；
3. 下载 `product-coach-skill-installable`；
4. 将下载的 ZIP 上传到 ChatGPT 的 `/skills` 页面；
5. 确认显示名称为“产品教练”，内部名称为 `product-coach-router`。

## 公开仓库与私有项目

仓库只包含通用规则、模板、脚本和虚构示例，不包含真实客户资料、公司内部 PRD、会议记录、代码包或个人信息。

优秀私有 PRD 可以在具体任务中临时用于校准结构、信息密度和表达方式，但不会自动写入公开仓库，也不会成为限制未来质量的固定范本。
