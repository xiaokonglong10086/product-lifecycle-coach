# 概念方案与方向确认

概念方案把“我们发现了什么问题”转成“准备采用什么产品方式解决”。它位于调研与落地 PRD 之间。

只有重大方向还没确定时才使用。小范围页面、文案、字段和局部机制调整，直接写变更说明，不重新做概念文档。

## 1. 什么时候需要

- 新产品或重大新模块；
- 产品核心机制仍有多种可能；
- 目标用户、问题、价值或运营方式发生变化；
- 用户希望先讨论产品思路，再进入详细需求；
- 当前 MVP 已经不能验证原来的假设。

## 2. 先读取什么

按可用性读取：
- 用户访谈、会议记录和调研证据；
- 当前工作流程、痛点和替代方案；
- 商业目标和公司资源；
- 竞品或相似做法；
- 当前产品、截图、代码和历史决定；
- 技术可行性和风险；
- 用户提出过的想法与已否定方向。

重复提到不等于已经确认。缺少证据时明确标注。

## 3. 工作步骤

1. 说清用户、场景、当前做法、问题后果和目标结果；
2. 找到最危险的假设，以及最强的“不应该做”理由；
3. 方向未固定时提出 2—3 个机制明显不同的方案；
4. 比较用户价值、可用性、可实现性、商业可行性、运营成本、信任风险、可逆性和学习速度；
5. 推荐一个方案，并说明其他方案为什么暂缓或放弃；
6. 画出核心闭环：触发、角色、关键动作、价值时刻、产出、人工控制和反馈；
7. 用一个真实端到端小闭环定义 MVP，而不是按页面数量删减；
8. 使用 `assets/templates/CONCEPT.md` 形成概念文档；
9. 检查方向是否一致、是否会导致明显不可实现问题；
10. 获得用户明确确认，或记录修改意见。

## 4. 概念文档结构

### 文档信息
- 状态、版本和日期；
- 负责人和确认人；
- 来源证据和上一版方向；
- 本次变化。

### 1. 产品背景
- 公司和业务背景；
- 为什么现在做；
- 当前处于原型、MVP、试点还是产品化阶段。

### 2. 目标用户与问题
- 使用者、购买者、运营者、管理者和受影响角色；
- 真实场景和当前流程；
- 问题、后果、频率和严重性；
- 当前替代方式及其不足；
- 支持和反对该判断的证据。

### 3. 产品主张
- 一句话产品思路；
- 用户使用前后发生什么变化；
- 为什么该机制可能创造价值；
- 与公司方向和替代方案的差异。

### 4. 候选方案比较
每个方案说明：
- 核心机制；
- 用户价值和见效时间；
- 对当前流程的改变；
- 数据、技术和运营要求；
- 失败方式和信任风险；
- 成本和可逆性；
- 选择、暂缓或拒绝原因。

### 5. 已选方案
- 核心用户闭环；
- 关键角色和动线；
- 系统边界；
- 主要页面/端；
- 概念层的重要规则；
- AI 与人工责任；
- 产品留下的数据或能力证据；
- 如何形成持续反馈。

### 6. 当前范围与明确不做
- 本轮 MVP/版本范围；
- 后续扩展；
- 明确不做；
- 试点允许的人工操作；
- 依赖和限制。

### 7. 成功、失败与学习
- 用户结果；
- 商业结果；
- 质量与信任信号；
- 防止副作用的护栏指标；
- 最危险假设；
- 试点最需要回答的问题；
- 何时停止、修改或扩大。

### 8. 风险与待确认
按需覆盖产品、采用、技术、运营、数据、AI、安全、法律和商业风险。

### 9. 确认记录
- 决定：确认 / 修改后再确认 / 暂缓 / 放弃；
- 确认人和日期；
- 被确认的范围；
- 不阻塞后续的遗留问题；
- 允许进入哪些后续工作。

## 5. 方案比较原则

候选方案必须在机制、运营方式、范围或 AI/人工分工上真的不同，不能只是界面颜色不同。

| 比较维度 | 方案 A | 方案 B | 方案 C |
|---|---|---|---|
| 用户价值 | | | |
| 见效时间 | | | |
| 对现有流程影响 | | | |
| 可实现性 | | | |
| 运营负担 | | | |
| 信任与失败风险 | | | |
| 验证速度 | | | |
| 可逆性 | | | |
| 长期价值 | | | |

分数可以辅助判断，但不能替代文字理由。

## 6. 原型、MVP、试点和正式版本

- **原型：** 验证用户是否理解、愿不愿意用、交互是否成立，不要求生产稳定性；
- **MVP：** 用真实端到端流程验证最危险的价值假设；
- **试点：** 在受控客户或人群中同时验证价值、部署和运营适配；
- **正式版本：** 可重复交付，有支持、监控、权限、治理和回退。

MVP 必须保留核心价值机制、真实入口到结果、关键异常、必要人工审核和数据采集。可延后额外人群、次要流程、广泛配置、平台化抽象和可人工替代的自动化。

## 7. 什么时候可以确认方向

只有以下条件基本成立时才进入落地 PRD：
- 用户和问题具体；
- 证据和不确定性可见；
- 已选机制能清楚解释；
- 记录了候选方案与取舍；
- 范围和明确不做清楚；
- MVP 仍然能验证核心假设；
- 有成功、失败和学习信号；
- 没有会让后续详细需求失去意义的重大未决问题。

必须获得明确确认，不能把沉默当同意。

## 8. 什么时候更新概念方案

以下变化需要重新确认：
- 目标用户改变；
- 问题或目标结果改变；
- 核心产品机制改变；
- MVP 不再验证同一个假设；
- 人工责任、治理或运营方式发生重大变化。

小 UI、文案、字段或实现调整不重开概念讨论，放入落地 PRD 或变更说明。

## 9. 常见错误

- 方向未确认就写详细 PRD；
- 第一想法直接当最终方案；
- 只描述页面，不解释产品机制；
- 按页面数量定义 MVP；
- 忽略购买者、管理员、运营者或受影响人；
- 没有证据就宣称差异化；
- 把重大方向问题写成 `[待补充]` 后继续开发。
