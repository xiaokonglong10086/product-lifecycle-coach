# 把一份过于混杂的产品文档拆清楚

当一份文档同时包含产品定位、页面、AI、API、数据库、QA、发布和路线图时，不要立刻重写。先做内容迁移表，保留有效决定，再把内容放到正确的工作文档。

对用户不使用 artifact migration 等术语，直接说“文档拆分与内容迁移”。

## 1. 先判断每段内容属于哪里

| 内容 | 应放入 |
|---|---|
| 用户、问题、价值、选定方案 | 概念/产品方向文档 |
| 用户动线、功能、交互、状态、边界、数据含义、文案 | 落地 PRD |
| 模型故事、Agent 工作流、Prompt 策略、评估和降级 | AI 产品专项规格 |
| 架构、数据库、接口、事件、部署、迁移 | 工程规格 |
| 需求 ID 到任务、代码和测试的关系 | 需求对应关系表 |
| 测试、试点、上线、监控和回退 | QA/发布计划 |
| 历史选择、废弃方案和原因 | 重要决定记录 |

## 2. 内容迁移表

每一章记录：
- 原文档和章节；
- 当前内容类型；
- 继续保留、修改、移动、合并还是废弃；
- 目标工作文档和章节；
- 已有需求 ID 是否保留；
- 与其他内容是否冲突；
- 谁确认；
- 当前状态。

## 3. 不要丢掉的东西

- 已经确认的产品规则；
- 用户明确否定过的方案；
- 稳定需求 ID；
- 真实代码现状和限制；
- 验收用例；
- AI 的人工确认、评估和降级要求；
- 版本、迁移和回退决定。

## 4. 需求 ID 如何处理

产品含义没有变化时，尽量保留旧 ID。确需更换时，记录：旧 ID、新 ID、更换原因、受影响任务、代码、测试、数据和发布内容。

这张关系表的作用不是增加流程，而是防止“PRD 改了，但 Codex 任务和 QA 还在执行旧规则”。

## 5. 拆分完成的条件

- 落地 PRD 可以从头读到尾，不被大量技术细节打断；
- AI、工程和 QA 的关键要求没有丢失；
- 所有已确认决定都有归属；
- 冲突和待确认项有负责人；
- 需求 ID 仍能对应到任务和测试；
- 团队知道下一步实际使用哪一份文档。

## 6. 独立产品经理的轻量做法

不必一次拆完整个历史文档。优先拆当前开发小闭环：
1. 从旧文档提取本轮相关段落；
2. 放入当前落地 PRD；
3. AI 或工程细节另存为短规格；
4. 保留旧文档为历史资料并标注“不再直接指导开发”；
5. 下一轮再处理其他模块。
