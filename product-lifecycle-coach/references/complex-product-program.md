# 复杂产品的全景与分步开发

复杂产品不要压进一份巨大 PRD，也不要同时推进所有端。先建立一张**产品全景与当前开发地图**，再一次完成一个端到端小闭环。

## 1. 产品全景需要回答什么

- 产品包含哪些业务模块和角色；
- 哪些规则是所有模块共享的；
- 哪些方向已经确认，哪些仍在试验；
- 当前代码已经做到哪里；
- 本期真正要验证的用户结果；
- 当前开发的小闭环是什么；
- 后续能力在什么条件下才启动。

专业方法里常把这张图叫 Program Map。对用户只说“产品全景与当前开发地图”。

## 2. 不同文档的职责

- **总产品 PRD：** 解释完整产品行为和模块关系；
- **当前模块/开发小闭环 PRD：** 解释本轮实现细节；
- **AI 专项规格：** 解释模型、Agent、人工控制和评估；
- **工程规格：** 解释代码、数据、接口和部署；
- **任务与 QA：** 解释怎么做、怎么验收。

只有某类内容真的存在时才创建对应文件。

## 3. 什么是端到端小闭环

专业上常叫纵向切片（vertical slice）。通俗理解是：

> 用户能从一个真实入口走到一个真实结果，虽然功能不多，但不是只有半个页面或孤立接口。

一个合格的小闭环应：
- 产生明确用户价值或验证关键假设；
- 跨越实现它所需的最少页面、数据和逻辑；
- 可独立演示、测试和验收；
- 有明确不做什么；
- 不依赖另一个尚未定义的大模块才能成立。

不合格的拆法：
- 只做所有页面外壳；
- 先做完整数据库但没有用户结果；
- 同时重构员工端、管理端和所有小游戏；
- 把一个大模块按前端/后端分成彼此不可验收的任务。

## 4. 选择当前小闭环

按以下顺序判断：
1. 是否验证当前最危险的产品假设；
2. 是否能给真实用户使用或给客户演示；
3. 是否依赖较少；
4. 是否可以在一到数轮 Codex 任务内完成；
5. 是否能产生数据或反馈，帮助决定下一步；
6. 是否不会破坏已确认的产品成果。

## 5. 复杂产品中的稳定规则与可变策略

### 尽量稳定
- 核心用户价值和业务闭环；
- 共享数据含义；
- 权限与安全原则；
- 学、测、用等关键证据的含义；
- 重要接口和版本规则。

### 可以迭代
- 页面布局和文案；
- 推荐算法和阈值；
- 奖励数值；
- 运营规则；
- 试点中的人工步骤；
- 非核心自动化。

稳定不等于永远不改。需要改时记录原因、影响和迁移方式。

## 7. 独立产品经理的执行限制

- 同一轮默认只推进 1 个小闭环；
- 活跃任务保持 1—3 个，最多 5 个；
- 优先选择能由你和 Codex 完成的低风险闭环；
- 需要生产级权限、数据和基础设施时，再把明确范围交给外包工程师；
- 外包前先有可演示原型、落地需求、验收和当前代码说明，避免按人天购买模糊开发。
