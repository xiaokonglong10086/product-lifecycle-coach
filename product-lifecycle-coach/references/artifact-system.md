# 工作文档与项目连续性

目标：让重要产品事实不只存在于聊天记录里，同时避免唯一产品经理被大量空文档拖累。

## 1. 默认最小集合

### Quick：小功能/修复
- `change-spec.md`：改什么、保持什么不变；
- `tasks.md`：Codex 任务；
- `qa.md`：验收。

### Standard：完整功能/模块
- `CURRENT_PRODUCT.md`：当前产品状态；
- `DECISIONS.md`：重要决定；
- `prd.md`；
- `tasks.md`；
- `qa.md`。

### Complex：复杂平台/多端/AI
按需要增加：
- 产品全景与当前开发地图；
- AI 专项规格；
- 工程说明；
- 发布和复盘。

不要一次性创建所有空文件。

## 2. 推荐目录

```text
product/
├── CURRENT_PRODUCT.md
├── DECISIONS.md
├── NEXT.md
├── specs/
│   └── <当前功能或开发切片>/
│       ├── concept.md          # 方向尚需确认时
│       ├── prd.md
│       ├── ai-spec.md          # AI 行为重要时
│       ├── experience.md       # UX 复杂时
│       ├── engineering.md      # 需要工程化时
│       ├── tasks.md
│       └── qa.md
└── releases/
    └── <版本>/
        ├── release-plan.md
        └── retrospective.md
```

## 3. 文件职责

### CURRENT_PRODUCT.md
只写用户真正需要的当前信息：
- 产品现在解决什么问题；
- 当前按什么方向和规则继续；
- 当前代码做到哪里；
- 本轮范围；
- 风险、阻塞和下一步；
- 你/Codex/外包分工。

### DECISIONS.md
追加记录重要决定：决定、原因、证据、拒绝方案、影响范围和确认人。

### NEXT.md
只保留当前 1-3 项任务，不做无限待办池。

### concept.md
选择和确认产品方向。只有确认后才授权详细落地 PRD。

### prd.md
产品行为说明。不能重新发明工程架构。

### ai-spec.md
AI/Agent 的模型故事、输入输出、人工确认、测试、失败和监控。

### engineering.md
只在技术复杂度需要时创建，说明架构、接口、数据、部署、安全和迁移。

### tasks.md / qa.md
任务与验收相互对应。

## 4. 局部变更

已确认产品进行局部修改时，用一份变更说明：

```text
现状
→ 改什么
→ 保持什么不变
→ 影响页面/数据/AI/代码
→ Codex/外包任务
→ 回归验收
→ 更新当前规则
```

禁止为了修改一个页面重写整份产品文档。

## 5. 来源优先级

1. 用户最新明确决定；
2. 最新已确认的概念/PRD；
3. 已记录的后续决定；
4. 当前代码事实；
5. 旧文档；
6. 推断或默认。

代码能证明“现在怎样实现”，不能自动证明“产品应该怎样设计”。

## 6. 状态与不确定性

状态：草稿、评审中、已确认、开发中、已验收、已发布、已废弃、已归档。

不确定性标记：
- `[已确认]`
- `[有来源]`
- `[推断]`
- `[假设]`
- `[建议]`
- `[待确认]`

落地 PRD 对真正阻塞的问题使用结构化 `[待补充｜...]`。

## 7. 项目恢复

继续旧项目时：
1. 读 `CURRENT_PRODUCT.md`；
2. 读当前 PRD 和重要决定；
3. 检查最新代码；
4. 对齐产品方向、规则和代码；
5. 只处理冲突和下一项不可逆决定；
6. 不自动重做调研或大 PRD。

## 8. 复杂项目的内部文件

必要时可保留以下内部文件名：
- `SOURCE_INVENTORY.md`：项目资料清单；
- `CURRENT_VERSION_ALIGNMENT.md`：当前版本对齐表；
- `PROGRAM_MAP.md`：产品全景与当前开发地图。

对用户展示时必须使用中文通俗名称，不要求用户理解 baseline、program map 等术语。
