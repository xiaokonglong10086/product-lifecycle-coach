# 产品全生命周期路由

本文件用于判断“现在最该做什么”。流程可以回退和迭代，但不能用后面的文档掩盖前面的关键决定没有完成。

## 1. 项目接入与恢复

适用：新想法、已有项目、散乱文档、继续上一轮、接手旧项目、不知道下一步。

动作：
1. 判断正在处理的是产品、版本、功能、缺陷、实验还是发布；
2. 读取已有资料、截图、代码和决定；
3. 明确当前产品方向、当前产品规则、当前代码现状；
4. 标出不一致、风险和阻塞；
5. 判断当前最早需要解决的问题。

输出给用户：
- 当前判断；
- 主要不一致；
- 本轮只做什么；
- 谁做；
- 完成标准。

## 2. 调研与问题验证

适用：目标用户、问题、市场、流程、价值、技术可行性或关键假设证据不足。

产出：
- 调研问题和计划；
- 证据与限制；
- 用户/相关角色洞察；
- 竞品和替代方案；
- 问题是否值得解决；
- 下一项证据行动。

进入下一步前必须满足：证据足以支持下一项决定，并且事实、推断和样本限制可见。

## 3. 产品策略与机会选择

适用：已经理解问题，但尚未决定服务谁、解决什么、为什么值得做、如何衡量。

产出：
- 产品简报；
- 目标用户；
- 问题与当前替代方式；
- 产品价值主张；
- 成功信号和反向指标；
- 继续、收窄、试验、暂停或停止建议。

## 4. 概念构建与确认

适用：问题值得解决，但产品机制、用户闭环、范围或方案尚未确认。

动作：
- 形成 2-3 个机制不同的方案；
- 比较价值、体验、成本、风险和学习速度；
- 推荐方案；
- 定义核心闭环、关键旅程、边界、AI/人工责任和 MVP；
- 记录拒绝方案及原因。

进入落地 PRD 前：用户或指定确认人必须明确认可方向。沉默或“先写写看”不算确认。

## 5. 落地需求

适用：方向已确认，需要让设计、Codex、开发和 QA 得到一致的产品行为说明。

产出：
- 固定八节落地 PRD；
- 主流程和异常分支；
- 功能树；
- 关键页面骨架；
- 核心功能的状态、边界、数据和验收；
- AI 功能引用；
- 待确认问题。

完成条件：没有聊天记录时，实施者也不会对关键产品规则产生不同理解。

## 6. UX 与视觉

适用：信息架构、布局、交互、视觉层级或多端行为会显著影响产品结果。

产出：
- 页面和信息架构；
- 关键旅程；
- 默认、空、加载、失败、无权限和恢复状态；
- 线框/原型；
- 视觉原则、组件和响应式规则。

## 7. 是否可以开始开发

适用：产品和体验方向已确认，需要给 Codex 或工程师实施。

检查：
- 产品规则是否清楚；
- 现有代码是否已查看；
- 数据、状态、权限和异常是否明确；
- AI 功能是否有输入输出、人工确认和测试；
- 是否明确哪些行为不能改；
- 是否有验收标准；
- 是用户 + Codex 能完成，还是需要技术复核/外包。

产出：
- 当前开发切片；
- Codex 任务；
- 必要的工程说明；
- 验收清单；
- 外包需求（如需要）。

## 8. 开发协作

循环：

```text
选择一个小闭环
→ 生成任务
→ Codex/工程师实现
→ 运行测试和构建
→ 产品经理按真实路径验收
→ 规格符合性检查
→ 修复
→ 记录完成与遗留
```

代码存在不等于完成。只有用户可观察行为满足需求，并有验证证据，任务才可关闭。

## 9. QA、试点与发布

产出：
- 产品验收；
- 回归测试；
- AI 评估（如适用）；
- 试点/UAT；
- 发布、监控、支持和回退计划；
- 发布决定。

生产级项目应由工程人员确认数据、安全、部署、监控和恢复。

## 10. 数据、反馈与迭代

产出：
- 指标和用户反馈；
- 原因分析；
- 假设是否成立；
- 修复、优化、扩展、转向或停止；
- 新一轮小任务；
- 产品经理经验沉淀。

每次发布至少回答：学到了什么、哪个判断改变、什么不该继续、下一项最大不确定性是什么。

## 11. 快速路径

### 小功能或 Bug
现状检查 → 变更说明 → Codex 任务 → 验收 → 回归。

### 已确认产品的局部修改
说明改什么和保持什么不变 → 检查影响 → 实现 → 验证 → 更新当前规则。

### 原型验证
明确假设 → 最小原型 → 测试对象和任务 → 观察标准 → 结论。原型不自动等于生产方案。

### 文档审查
明确文档用途 → 对照来源和决定 → 检查战略/结构/可实现性 → 直接修订或生成修复计划。

## 12. 复杂产品路径

复杂产品不要一次全面推进：

```text
当前版本对齐
→ 产品全景与当前开发地图
→ 选择一条端到端小闭环
→ PRD/UX/AI/开发任务
→ 实现和 QA
→ 获得证据
→ 再选择下一条
```

对用户只展示当前闭环和下一步，不要求用户维护过多方法论文件。

## 13. 产品成熟度路线

任何开发或发布任务先标明：
- 概念原型；
- 可交互 Demo；
- 内部可用版；
- 客户试点版；
- 正式生产版。

使用 `prototype-to-production.md`。当前阶段决定可以 Mock 什么、需要什么证据，以及是否必须由工程师承担责任。

## 14. 初创 B2B 快速路线

```text
客户问题与场景
→ 销售演示证明客户理解价值
→ 小范围真实试点证明用户和运营价值
→ 补齐生产工程条件
→ 正式上线与复制交付
```

演示反馈不能替代试点数据；试点成功也不能替代权限、安全、部署和恢复验证。
