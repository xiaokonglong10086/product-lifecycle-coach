# 工程规格、Codex 开发与外包交接

目标不是写最多的技术文档，而是让实现过程不需要临时猜产品规则。只有产品方向、落地需求和关键交互足够清楚后，才进入开发。

对用户输出时先说：**现在是否可以开始开发、先做哪一个小闭环、谁负责什么。** 技术细节放在后面或独立文件中。

## 1. 开始开发前需要具备什么

按任务风险选择必要输入：
- 已确认的产品方向和落地需求；
- 关键页面、状态和视觉规则；
- 明确不做的内容；
- 当前代码结构、现有行为和构建方式；
- 数据、权限、安全、部署和测试约束；
- 相关需求 ID 与验收标准。

代码可访问时先查看真实文件、依赖、测试和构建命令。不得凭空发明目录、组件或接口。

## 2. 工程规格写到什么程度

只写本轮实现真正需要的部分：
1. 本轮目标、范围、不做什么、来源需求；
2. 当前代码怎么工作，存在什么限制；
3. 本轮方案和模块边界；
4. 关键组件或服务分别负责什么；
5. 数据对象、状态流转和数据保存；
6. 接口、事件、错误和重试；
7. 权限、隐私、审计和安全；
8. AI 流程、模型输入输出、评估和降级（如适用）；
9. 性能、兼容、成本和监控；
10. 测试、发布、迁移和回退；
11. 需求如何对应到代码、任务和测试。

原型或低风险前端功能不需要机械补齐全部 11 项。生产数据、多租户、敏感信息和高影响 AI 则不能省略。

## 3. 代码和模块边界

每个重要模块回答：
- 它负责什么；
- 读取什么；
- 输出什么；
- 可能在哪里失败；
- 如何测试；
- 如何修改而不破坏其他模块。

优先沿用当前项目的合理模式。只做服务本轮目标的重构，不借机全面整理代码。

## 4. 数据与状态

落地 PRD 定义数据对用户和业务的含义；工程规格决定数据库、缓存、队列和索引等实现方式，但不得改变产品含义。

按需定义：
- 核心对象和稳定 ID；
- 数据由谁维护，哪里是唯一可信来源；
- 状态如何变化；
- 重复操作如何避免重复写入或发奖；
- 并发冲突如何处理；
- 历史版本、审计和迁移；
- 删除、归档和保留；
- 部分失败后的补偿和对账。

超过两个重要状态的流程，优先画状态图。

## 5. 接口与集成

接口、事件或第三方连接按需说明：
- 谁在什么条件下触发；
- 输入和输出；
- 校验与权限；
- 错误如何映射成用户提示；
- 超时、重试、退避和重复请求；
- 顺序与版本兼容；
- 日志、监控和审计。

## 6. AI 工程补充

AI 功能至少说明：
- 哪些步骤由固定代码完成，哪些交给模型；
- 输入来源、优先级和可信度；
- 结构化输入输出；
- 模型、Prompt、知识库和工具版本；
- 延迟、Token 和成本上限；
- 依据、不确定性和拒绝回答；
- 人工确认和修正；
- 标准测试样例、指标和阻断错误；
- 失败、重试、部分结果和人工替代；
- 提示注入、隐私和数据泄露防护；
- 线上 bad case 回流、复现和审计。

## 7. 拆成 Codex 可以完成的小任务

先确认本轮最小可演示结果，再拆任务。每个任务必须包含：
- 目标和关联需求；
- 修改范围和禁止修改项；
- 已决定的产品规则；
- 可能涉及的文件；
- 实现步骤；
- 测试与构建命令；
- 验收标准；
- 数据或迁移影响；
- 风险和回退；
- Codex 最终必须汇报的修改文件、测试结果和遗留问题。

拒绝“增加校验”“处理异常”“补测试”这类无法验收的任务描述。

## 8. 默认 Codex 提示词结构

1. 角色设定；
2. 项目背景；
3. 当前代码行为和问题；
4. 本轮唯一目标；
5. 允许修改范围；
6. 禁止项；
7. 必须实现的规则；
8. 测试、类型检查和构建命令；
9. 验收标准；
10. 输出格式：修改文件、实现说明、验证结果、遗留风险。

要求 Codex 先读代码再修改；遇到缺失的产品决定时停止并说明，不能自行补规则。

## 9. 一轮开发的正确节奏

1. 选一个可演示的小闭环；
2. 确认范围和不能改变的内容；
3. Codex 完成一个有边界的任务；
4. 运行测试、类型检查和构建；
5. 查看真实页面、数据和代码差异；
6. 先检查是否符合产品要求，再检查代码质量；
7. 修复问题；
8. 更新当前产品状态、任务和变更记录；
9. 再进入下一任务。

## 10. 谁来做

### 你来做
- 用户、问题、价值、范围和优先级；
- 用户可见规则、状态、文案和取舍；
- 原型和验收；
- 客户沟通、试点和发布决定；
- 需要人工承担的业务、伦理或政策决定。

### Codex 做
- 读取代码并说明现状；
- 有边界的前端、逻辑和轻量后端实现；
- 重构、测试、类型检查、构建和文档；
- 复现缺陷、收集证据和提出技术方案；
- 在本地或受控环境完成原型和 Demo。

### Codex 做、外包工程师复核
- 需要真实数据库但影响范围有限；
- 普通第三方接口接入；
- 有权限控制但不涉及高敏感数据；
- 需要部署但尚未形成高并发或高可用承诺；
- AI 流程将进入客户试点，但暂不自动作出高影响决定。

### 外包工程师负责
- 登录、权限、SSO、多租户隔离；
- 敏感数据、加密、安全加固和合规；
- 正式数据库迁移、备份、恢复和灾难恢复；
- 生产部署、监控、告警、运维和事故处理；
- 支付、高并发、实时协作或复杂集成；
- 文件扫描、转码和大文件服务；
- AI 自动执行不可逆动作或影响正式评价、薪酬、录用、淘汰等决定。

## 11. 是否可以开始开发

以下任一项成立，应暂停并先补产品决定：
- 关键需求仍有多种解释；
- 核心页面和状态未确认；
- 业务规则、权限或异常缺失；
- 没看过当前代码就提出修改；
- 核心需求没有对应任务和验收；
- 数据迁移、安全、权限或回退方案缺失；
- 一个任务混入多个无关系统；
- 提示 Codex 自己决定产品方向。

## 12. 代码压缩包和构建证据

ZIP 中包含 `node_modules` 时，只能把它当作源码快照。Rollup、Rolldown、esbuild、Sharp 等原生依赖可能只适用于打包者的操作系统。

记录：
- 是否包含依赖；
- 原操作系统与目标环境；
- lockfile 和包管理器版本；
- 目标环境重新安装依赖的结果；
- 类型检查、测试和构建结果；
- 失败属于源码、依赖、环境还是打包。

不能因为 Windows 依赖在 Linux 失败就断言源码无法构建，也不能因为旧 `node_modules` 恰好可用就声称正式构建通过。

## 13. Codex 任务预检查

每张任务优先使用 `assets/templates/CODEX_TASK.md`。提交前检查：
- 是否只有一个可验收结果；
- 是否要求先读真实代码；
- 是否明确允许修改和禁止修改；
- 是否写清默认、加载、空、失败、重复和返回；
- 是否有可观察验收场景；
- 是否要求运行测试、类型检查和构建；
- 是否要求汇报修改文件、未解决问题和未授权改动。

运行：

```bash
python scripts/audit_codex_task.py path/to/task.md --strict
```

复杂修改前建议新建分支或可恢复备份。不要要求 Codex 在没有现状测试的情况下大范围重构。

## 14. 视觉和交互验收证据

涉及前端时，除构建结果外至少保留：
- 核心页面截图或录屏；
- 移动端目标宽度；
- 主流程和异常流程；
- 返回、取消、加载、空和失败状态；
- 修改前后差异；
- 未破坏旧功能的回归路径。
