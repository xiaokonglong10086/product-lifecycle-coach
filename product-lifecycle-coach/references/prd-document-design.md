# 产品功能需求文档的阅读与版式合同

## 1. 阅读目标

读者应当先看到完整产品，再进入单个流程和功能细节。正文顺序遵循：产品全貌 → 完整流程 → 内部逻辑 → 通用规则 → 业务功能 → 跨功能规则 → 质量与约束。

## 2. 信息层级

- 一级标题：产品的重要组成部分；
- 二级标题：页面、任务或核心功能；
- 三级标题：字段组、子机制或需要独立定位的细节；
- 不使用五六层目录。

## 3. 流程图

- 完整产品流程单独占据足够页面空间，保证文字可读；
- 模块流程可以简化，但必须保留异常和返回路径；
- 流程图之后只写必要的结果说明，不重复逐节点解释；
- 用户流程与内部运作流程分开呈现。

## 4. 表格与正文

- 字段、状态、优先级、错误等级、数据去向和指标适合表格；
- 操作流程、AI 判断规则、覆盖逻辑和模块协作使用短段落或编号步骤；
- 不把复杂功能压成一张三四行概括表；
- 同一规则只在最合适的位置出现一次。

## 5. 版式

- 正文段落尽量不超过 5 行；长机制使用小标题和编号；
- 重要功能字段不得因为分页而丢失表头；
- 页面线框和流程图必须在渲染后检查可读性；
- 正式文档不得带修订过程、旧版本对照或内部审计语言；
- 文档标题使用“产品功能需求文档”，除非用户明确要求其他类型。

## 6. 补充文档

工程规格、AI 功能规格、QA 和开发任务使用独立文件。它们引用主 PRD 的功能名称，不在主 PRD 内反复展开实现细节。
