# 方法来源、适配与维护记录

本 Skill 是机制级综合，不是对单一仓库的直接复制。外部来源用于方法参考，用户确认的工作方式决定最终行为。

## 1. BMad Method

来源：`bmad-code-org/BMAD-METHOD`（MIT）。

采用：
- 从调研、Product Brief、PRD、UX、架构、开发、审查到复盘的生命周期；
- 按复杂度和风险调整深度；
- Create / Update / Validate 意图；
- 快速与教练路径；
- 决策日志、来源对账和 reviewer gate；
- 上游 BMAD 允许 PRD 结构按风险调整；本 Skill 仅吸收其内容深度和审查机制，不采用其自由目录。用户的落地 PRD 固定为八段阅读合同。

修改：
- 本 Skill 不只“提取用户愿景”，还必须主动批判和提出更优方向；
- 将 BMad 运行时脚本和多 Agent 依赖改为跨 ChatGPT/Codex/GitHub 的轻量 artifact。

## 2. GitHub Spec Kit

来源：`github/spec-kit`（MIT）。

采用：
- 规格作为长期合同；
- what/why 与技术方案分离；
- 可独立测试的用户场景；
- 稳定、可测试需求；
- 只澄清高影响歧义；
- 规格 checklist、自修复、跨 artifact 一致性和收敛。

修改：
- 补充企业产品所需的业务规则、权限、状态、数据、运营和 AI 治理；
- 门禁用于保护质量，不作为机械瀑布流程。

## 3. OpenSpec

来源：`Fission-AI/OpenSpec`（MIT）。

采用：
- 变更提案、增量规格、设计、任务、实现、验证、同步和归档；
- 行动式而非僵硬阶段式迭代；
- 项目上下文和 artifact 特定规则；
- Update 与 New Change 的判断。

修改：
- 将变更治理向上扩展到调研、概念、UX、试点和产品学习。

## 4. Superpowers

来源：`obra/superpowers`（MIT）。

采用：
- 创作和实现前先理解现状；
- 比较 2-3 个方案；
- 设计确认后再开发；
- 大项目拆分；
- 面向零上下文执行者的计划；
- 小步交付、规格审查优先于代码质量审查、完成必须有证据。

修改：
- 不对所有产品、原型和遗留项目机械强制 TDD；
- 紧急执行允许批量询问少数阻塞问题。

## 5. Agent OS

来源：`buildermethods/agent-os`（MIT）。

采用：
- 规划前读取产品背景、代码规范、参考实现和视觉材料；
- 保存 shaping 决策；
- 只注入与当前任务相关的标准；
- 让规格与现有代码库保持一致。

## 6. Vibe-Coding Prompt Template

来源：`KhazP/vibe-coding-prompt-template`（MIT）。

采用：
- 对新手友好的研究、PRD、技术方案、Agent 配置和开发路径；
- 简单的 Plan → Execute → Verify 循环。

未作为主框架：
- 对复杂企业、AI、多角色和长期产品的业务规则、变更、QA 和运营深度不足。

## 7. 用户确认的落地需求结构

采用用户提供的核心结构：
- 概念版明确确认后才能进入落地版；
- 用户场景、Mermaid 流程、优先级功能树；
- 关键页面 ASCII 线框；
- 核心功能逐项写触发、交互、状态、边界和数据；
- 产品文案、非功能需求和结构化待补充。

适配原则：
- 文档信息放在未编号封面区，不增加“0. 文档控制”；
- 当前范围、非目标和约束收敛在第 1 节；
- 跨功能规则尽量归入相关核心功能，无法归属时建立独立规则字典；
- 验收、追踪、工程、发布和回滚进入伴随文档；
- AI 行为进入独立 AI 产品专项规格，不增加第 9 节。

## 8. AI 产品 PRD 示例（DeerFlow）

来源：张佳《AI产品需求文档框架-以 Deer-Flow 为例》，基于开源 DeerFlow 项目反向整理的教学型 PRD。

采用：
- 用户故事与模型故事分离；
- 用户旅程与 Agent 工作流分离；
- 每个 Agent 明确上下文、能力、工具、输出、约束和决策规则；
- 人在环中的计划审核与修改；
- Prompt 章节写设计策略、输出控制和示例，而不是用完整 Prompt 替代产品需求；
- 标准测试样例集、确定性测试与概率性评估并行；
- 幻觉、引用、完整性、工具调用、延迟、成本等可测指标；
- Agent 失败、重试、跳过、备用工具和终止流程；
- AI 版本规划和线上 bad case 回流。

修改：
- 不照搬该文档中面向成熟团队的大样本量和完整多 Agent 结构；
- 按初创公司和唯一产品经理的实际能力设置 Level A/B/C 三档规格；
- 原型阶段允许 10-20 条标准样例，试点阶段再扩大；
- 高风险确认点不得因超时自动通过；
- 不把提示词实现细节塞进主 PRD。

## 9. 独立产品经理与初创公司适配

采用用户实际工作方式：
- 默认由用户承担产品、原型、Codex 协作、验收和客户沟通；
- 把“基线、门禁、artifact、traceability”等术语转换为可理解的中文工作语言；
- 每轮明确你、Codex、外包工程师的分工；
- 默认只保留当前产品状态、重要决定和下一步任务三类长期文件；
- 生产级权限、数据、安全、基础设施和高影响 AI 决策建议由工程人员负责；
- 专业严谨性通过状态、边界、数据、验收和回归保持，而不是通过增加文件数量体现。

## 10. 维护规则

更新本 Skill 时：
1. 从真实使用中确认重复失败；
2. 找到负责该失败的指令、模板或脚本；
3. 与当前上游框架比较；
4. 只修改最小但足够的部分；
5. 至少用一个复杂产品和一个 Quick 变更测试；
6. 记录来源、变更和原因；
7. 检查模板、脚本和 SKILL.md 是否相互一致。

不要因为方法流行而添加规则。每条规则必须解决观察到的问题或保护高代价决策。
