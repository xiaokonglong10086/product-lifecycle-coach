# 独立产品经理任务手册

本文件把产品方法转成“遇到什么情况，下一步怎么做”。默认先完成一个可验证结果，不要求用户先理解整套方法论。

## 1. 使用原则

- 先判断用户当前卡在哪一个具体任务，不从产品生命周期第一页重新开始。
- 能从现有资料推断的内容直接完成；只有涉及方向、范围、风险和取舍时才请用户决定。
- 每次最多推进 1 个主任务和 2 个辅助任务。
- 提问时一次只问 1 个高价值问题，并提供 2-4 个可选方向和推荐意见。
- 正式文档前先给一屏行动摘要，说明为什么做、谁做、完成后得到什么。

## 2. 八类常见场景

### 场景 A：我只有一个想法

目标：判断是否值得继续，而不是马上写 PRD。

步骤：
1. 说清谁在什么场景遇到什么问题；
2. 找出现有替代做法；
3. 识别最危险的 1-3 个假设；
4. 用访谈、竞品、流程观察或原型验证；
5. 决定继续、收窄、改方向或停止。

最小产出：一页产品简报 + 下一项验证任务。

### 场景 B：方向已经想清楚，准备写 PRD

目标：把用户可见行为写清楚，不替开发决定架构。

步骤：
1. 确认产品方向已明确认可；
2. 确认本期范围和明确不做；
3. 画主流程和异常分支；
4. 列功能树；
5. 逐个核心功能写触发、交互、状态、边界、数据和验收；
6. AI 功能另写 AI 功能卡或专项规格。

最小产出：固定八节落地 PRD。

### 场景 C：已经有 Demo，不知道下一步

目标：区分“能演示”和“能试点/上线”。

步骤：
1. 记录 Demo 实际能完成的用户闭环；
2. 标出 Mock、硬编码、无法恢复和未接真实数据的部分；
3. 判断当前属于演示、内部可用、客户试点还是生产；
4. 选择下一条端到端小闭环；
5. 决定用户 + Codex、技术复核或外包。

最小产出：当前产品状态 + 产品成熟度检查 + 下一步任务卡。

### 场景 D：要让 Codex 开发

目标：让 Codex 不需要自行猜产品规则。

步骤：
1. 先读真实代码和当前行为；
2. 选一个唯一可验收结果；
3. 写明允许修改与禁止修改；
4. 写清状态、异常、数据和验收；
5. 要求测试、类型检查、构建和修改文件说明；
6. 产品经理按真实路径验收。

最小产出：一张 Codex 开发任务卡。

### 场景 E：Codex 做完了，要验收

目标：验证用户结果，不以“代码完成”为完成。

步骤：
1. 核对修改文件和未授权改动；
2. 运行测试、类型检查和构建；
3. 走主流程、异常、返回和恢复；
4. 检查移动端、空状态、失败状态和旧功能；
5. 对照验收标准记录证据；
6. 只生成下一轮最小修复任务。

最小产出：QA 报告 + 修复任务。

### 场景 F：要做 AI 功能

目标：先决定 AI 的责任边界，再考虑模型和 Prompt。

步骤：
1. 判断规则/搜索/模板能否解决；
2. 选择 Level A/B/C；
3. 写用户任务、允许决定和禁止决定；
4. 写输入、输出、人工确认和失败替代；
5. Level B/C 再写模型故事和工作流；
6. 准备真实标准样例，运行评估后再试点。

最小产出：AI 功能卡或 AI 专项规格。

### 场景 G：要给客户演示或试点

目标：区分“讲清价值”和“验证真实使用”。

步骤：
1. 明确是销售演示还是客户试点；
2. 演示只保留最有说服力的闭环；
3. 试点明确真实用户、数据、运营和支持；
4. 设定成功、停止和扩大条件；
5. 试点结束用证据决定下一版。

最小产出：演示脚本或客户试点计划。

### 场景 H：需要外包工程师

目标：把必须工程化的部分交出去，同时保留产品控制权。

步骤：
1. 说明业务目标和用户流程；
2. 划清本期范围、现有代码和必须保留内容；
3. 写角色权限、数据、安全、环境和非功能要求；
4. 要求外包提交方案、风险、排期和维护方式；
5. 用分阶段交付和可验收里程碑付款；
6. 确认源码、账号、部署文档和知识产权归属。

最小产出：外包工程交接说明。

## 3. 时间投入建议

时间只用于控制范围，不作为质量借口。

- 30-60 分钟：小功能判断、变更说明、单张任务卡；
- 半天：概念比较、模块 PRD、一次代码审查或 QA；
- 1-2 天：完整功能 PRD、AI Level B 规格、客户试点设计；
- 多轮推进：复杂平台、跨端共享规则、生产工程和高影响 AI。

若任务无法在当前时间内完成，缩小目标，不删除关键状态、异常、权限和验收。
