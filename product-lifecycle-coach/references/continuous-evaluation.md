# Skill 持续回归与优化

把 Skill 当作产品。每次优化必须来自真实使用问题，并通过重复测试验证，而不是只凭感觉改文案。

## 1. 固定测试案例

至少保留以下非机密案例：
1. 证据不足的新想法；
2. 已确认方向，需要写落地 PRD；
3. 多文档、多版本、代码并存的复杂产品；
4. 混合 PRD，需要拆分；
5. 需要检查代码与产品要求差异；
6. AI/Agent 功能，需要模型故事、人工确认和评估；
7. 小功能，需要 Codex 任务、QA 和发布；
8. 初创公司唯一产品经理，需要判断自己/Codex/外包分工。

不要把用户的机密项目文件打包进 Skill。

## 2. 评分维度

每项 0-2 分，并给证据：
- 当前产品方向、规则和代码是否对齐；
- 生命周期判断是否正确；
- 已确认决定是否被保护；
- 是否敢于指出产品问题并给替代方案；
- PRD 是否完整、可读、符合格式；
- AI 产品是否有模型故事、测试、失败和人工责任；
- 是否检查真实代码和运行环境；
- 当前开发切片是否足够小且端到端；
- 需求、任务和测试是否对应；
- QA、试点和发布是否可靠；
- 新手是否能理解；
- 是否明确用户/Codex/外包分工；
- 文档数量是否克制；
- 输出是否先给结论和下一步。

以下任一项为 0，不能升级版本：
- 方向/规则/代码对齐错误；
- 未确认方向却直接写落地 PRD；
- 高影响 AI 无人工确认；
- 发布无验证证据；
- 用户看不懂下一步或不知道谁做。

## 3. 回归流程

每次修改：
1. 记录真实问题；
2. 找到负责该问题的指令、模板或脚本；
3. 只修改最小但足够的部分；
4. 运行确定性脚本；
5. 重跑暴露问题的案例；
6. 重跑至少一个旧案例；
7. 比较是否引入新问题；
8. 更新评估报告；
9. 验证后重新打包完整 `skill.zip`。

## 4. 必测内容

### PRD
- 固定八节；
- 核心功能包含状态、边界、数据和验收；
- AI/工程内容没有挤爆主 PRD。

### AI 规格
- 规格深度选择正确；
- Level B/C 包含模型故事；
- Level C 包含 Agent 工作流；
- 有标准测试样例；
- 确定性测试和概率性评估分开；
- 有失败、重试、降级、成本和监控。

### 独立产品经理体验
- 首屏能知道当前判断；
- 专业词有通俗解释；
- 最多 1-3 个默认任务；
- 每个任务明确谁做；
- 外包建议有明确触发理由；
- 不创建无用空文档。

## 5. 评估报告

使用 `SKILL_EVAL_REPORT.md`，记录：
- Skill 版本和日期；
- 真实案例；
- 预期与实际判断；
- 评分和证据；
- 暴露问题；
- 本轮修改；
- 脚本/构建结果；
- 仍有限制；
- 下一次回归案例。

## 6. V0.6 新增回归场景

9. 简单 Level A AI 助手，必须能用一页功能卡完成，不能被迫填写多 Agent 大文档；
10. 可交互 Demo 准备进入客户试点，必须识别真实数据、人工运营和工程责任缺口；
11. Codex 开发任务必须能被脚本阻止“全面优化”与缺少验收/构建的情况；
12. 外包工程交接必须覆盖源码、账号、数据、安全、里程碑、质保和维护；
13. 用户需要做决定时，输出必须提供选项、取舍和建议，而不是开放式追问。

## 7. 自动自检

每次修改后运行：

```bash
python scripts/self_test.py
```

该脚本至少验证：
- 简单 AI 功能卡能通过，空泛 AI 需求会失败；
- 有边界的 Codex 任务能通过，“全面优化/自由发挥”会失败；
- 完整外包交接能通过，缺少源码、账号、数据安全和里程碑会失败。
