# 产品教练 Skill

`product-coach-router` 是一个帮助产品经理持续推进真实产品项目的 ChatGPT Skill。

它的核心不是“调度六个模块”，也不是“生成一堆产品文档”，而是解决一个更根本的问题：

> 在会议、文档、原型、代码、反馈和多轮 AI 协作不断变化的情况下，仍然让所有人知道当前产品是什么、哪些决定已经确认、这轮最应该解决什么，以及怎样把它真正做出来并验证。

## 核心工作

产品教练会围绕四件事持续工作：

1. **理解当前产品**：读取已有资料和实现，恢复当前方向、产品规则、代码现状与冲突；
2. **保护产品连续性**：区分事实、用户决定、AI 建议和待确认问题，防止改文档或改代码时悄悄改变产品；
3. **完成当前最重要的工作**：根据实际需要完成调研、方向判断、设计原则、PRD、开发任务、实现验收或迭代方案；
4. **用结果验证进展**：检查真实文件、构建、运行和用户操作，不用“已经完成”的文字替代证据。

## 如何使用

用户不需要了解内部模块，也不需要先判断自己处于产品生命周期的哪个阶段。

把会议记录、PRD、截图、原型、代码和反馈放入当前项目，然后直接描述本轮目标。例如：

```text
调用产品教练。先读完当前资料，告诉我这个产品已经确定了什么、哪里存在冲突、现在最应该先解决什么，并直接完成这一步。
```

```text
基于当前已确认设计生成正式 PRD。不要新增未经确认的角色、流程和权限，信息不足时明确标记待确认。
```

```text
基于 PRD 和现有代码生成 Codex 开发任务。我要能够实际操作和验证主要功能、异常情况和跨页面结果，不要只生成静态页面。
```

```text
实际验收当前项目，区分产品设计问题、实现偏差、体验问题和上线差距，并生成下一轮修复任务。
```

## 每轮应该呈现什么

产品教练对用户的输出应优先说明：

- 当前判断；
- 判断依据和来源冲突；
- 本轮真正需要解决的问题；
- 已生成、修改或审查的真实产物；
- 哪些事项仍需用户决定；
- 下一步最值得做什么。

内部路由、模块名称、质量等级和审查脚本属于执行手段，不应占据用户表达的中心。

## 核心理念

- 从当前产品继续，不机械重走完整流程；
- AI 主动分析和推荐，但重要产品决定由用户确认；
- 只维护一套当前产品事实和正式决定；
- 文档服务于理解、实现和验收，而不是为了显得专业；
- 产品必须通过真实使用验证，不能用静态展示冒充完成；
- 保留已经做对的内容，只修改真正受新证据影响的部分；
- 在完成真实工作的同时，帮助用户理解关键产品判断。

## 内部结构

一个用户入口会按当前任务加载对应专业规则：

- 需求发现与方向判断；
- 长期产品设计原则；
- 正式产品功能 PRD；
- 产品设计验证与 Codex 开发；
- 实现 QA；
- 反馈、迭代、试点与上线。

工程规格、AI 功能规格、模拟数据和文档交付作为共享能力按需使用，不会反向修改用户已经确认的产品设计。

## 安装与维护

通过仓库 Actions 中的 **Build Product Coach Skill** 下载最新成功构建工件 `product-coach-skill-installable`，上传到 ChatGPT `/skills`。

本地验证：

```bash
python scripts/validate_skill.py .
python scripts/self_test.py
python scripts/package_skill.py
```

公开仓库不保存真实客户资料或完整私有 PRD。私有优秀文档只用于具体任务的临时校准，不自动写回仓库，也不限制未来输出质量。
