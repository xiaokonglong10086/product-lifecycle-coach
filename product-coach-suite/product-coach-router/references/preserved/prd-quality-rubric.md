# PRD 质量评分规约

本规约代替把某一份历史 PRD 固化为唯一黄金样本。样本只提供已验证方法，不定义质量上限。

## 评分方式

每项按 0-4 分评分：
- 0：缺失或严重错误；
- 1：存在但无法支持开发或验收；
- 2：基本合格，仍有明显模糊处；
- 3：成熟、清楚、可实现、可验收；
- 4：在不改变产品事实的前提下显著提升理解效率和决策质量。

交付门槛：
- 来源忠实度、功能展开深度、流程完整度均不得低于 3；
- 任一维度不得低于 2；
- 总平均分不得低于 3；
- 结构审查、来源覆盖审查与 DOCX 视觉检查必须通过。

## 十项质量维度

### 1. 产品画面完整度
读者能否理解产品服务谁、在什么场景使用、页面如何连接、任务如何完成、结果流向哪里。

### 2. 来源忠实度
是否完整保留用户已确认的页面、规则、状态、异常、数据去向和禁止项；是否把推断明确标为待确认。

### 3. 功能展开深度
复杂功能是否写到入口、布局、字段、完整操作、反馈、状态、异常恢复、结果去向、跨端影响和验收，而非模块名称或摘要。

### 4. 流程完整度
是否覆盖主流程、关键判断、失败、部分失败、重试、返回修复、人工确认、正式写入和闭环回流。

### 5. 信息密度
是否具体而不重复；是否避免空洞愿景、模板化废话和同一规则在多章反复出现。

### 6. 阅读连续性
一个完整机制是否集中展开；是否先给页面布局和整体流程，再写局部规则；表格是否只用于适合对照的信息。

### 7. 产品语言
是否先说明用户看到什么、做什么、得到什么，再说明系统如何处理；是否避免工程实现淹没产品行为。

### 8. 验收可执行性
验收标准是否可观察、可操作、可复现，是否覆盖成功和失败状态，而非“体验良好”“准确率高”等模糊表述。

### 9. 工程噪音控制
是否保留必要的产品内部运作、异步、人工确认和数据流规则，同时避免 API 路径、数据库表、DDL、完整 Prompt 和基础设施细节占据正文。

### 10. 待确认管理
是否说明未知问题为何不能确定、需要谁决定、有哪些选项、临时处理是什么，而不是编造规则或留下无上下文的“待补充”。

## 动态改进规则

可以突破既有示例的结构与表达，只要新方法：
- 更适合当前产品复杂度；
- 提高质量规约得分；
- 不降低来源忠实度、功能深度、异常恢复和人工确认边界；
- 不把新的表达方式变成未经确认的产品规则。

不得为了模仿任何历史 PRD 而降低当前文档质量。
