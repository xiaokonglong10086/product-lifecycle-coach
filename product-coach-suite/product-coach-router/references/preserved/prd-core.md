# 产品功能 PRD 专项（完整保留版）

只完成一件事：把权威产品资料整理成一份唯一、完整、可读、可实现、可验收的《产品功能需求文档》。不要输出产品教练分析，不做生命周期治理，不拆成 PRD 与 FRD。

## 1. 首先冻结产品事实

1. 读取用户指定的全部权威来源：现有 PRD、会议记录、截图、原型、Demo、代码说明和本轮确认。对长期项目，先读取已有 `DESIGN_PRINCIPLES.md` 与 `DECISIONS.md`。
2. 只有出现新的长期判断、理念冲突、多次反馈形成稳定规律或用户明确要求时，才调用设计理念模块更新；不得每次生成 PRD 都重新提炼理念。
3. 按以下优先级处理冲突：用户本轮明确确认；已确认长期理念；用户指定主文档；用户认可原型、截图和 Demo；后续会议明确决定；早期资料；AI 推导。
4. 在内部建立规则清单，至少提取页面、布局、入口、字段、操作顺序、状态、异常、恢复、AI 行为、数据来源、数据去向、质量要求和明确不做。
5. 不得把格式优化变成产品重设计。未经确认的新角色、新审批、新权限、新模板层级和新后台能力只能先提出。
6. 设计理念约束长期方向，但不能替代具体产品规则，也不能被扩写成用户未确认的新功能。
7. 信息不足时写“待确认”，不得猜测补齐。

同时读取：
- `prd-authority-and-freeze.md`
- `prd-contract.md`
- `prd-flow-and-function-standard.md`
- `prd-gold-pattern.md`
- `docx-quality-gate.md`

## 2. 只生成一份主 PRD

默认文件名：`[产品名称]_产品功能需求文档.docx`

正式文档只能描述当前产品是什么、用户如何使用、产品内部如何运作、功能如何呈现和反馈、异常如何恢复、结果写到哪里、什么叫做对。

正式文档禁止出现：
- 产品教练、教练结论、教练建议；
- 文档版本、PRD Vx、V0.x；
- P0、P0A、P0B、P1、P2；
- 基线、本版、上一版、原版本、相比旧版、版本变更；
- 第 0 章；
- FRD、以 FRD 为基线、PRD/FRD 分工；
- 开发批次、路线图、生命周期治理；
- AI 如何生成本文件的说明。

只有用户明确要求时才允许出现其中某项。

## 3. 文档必须具备三类真实流程图

### 3.1 完整产品总流程图

从产品入口开始，覆盖全部主要任务分支、多个角色或端的衔接、关键判断、失败、部分失败、重试、返回修复、人工确认、正式写入、最终结果和闭环回流。

一排阶段卡片、单向箭头或模块关系图不能替代完整产品总流程。

### 3.2 核心用户流程图

每条主要用户任务单独展开，体现用户可见操作、反馈、异常和恢复。多角色产品至少分别绘制主要角色流程。

### 3.3 产品内部运作图

至少体现：用户操作 → 任务登记 → 校验、排队或异步处理 → 外部服务或 AI 子任务 → 部分失败隔离 → 待确认草稿 → 通知召回 → 人工确认 → 正式写入 → 跨端更新 → 质量回流。

DOCX 中必须真正嵌入图，并为每张图设置标题或替代文本。不得写“此处插入流程图”。

## 4. 功能细节是正文主体

按用户任务顺序展开业务模块。每个复杂功能必须完整说明：

1. 功能作用；
2. 页面位置、入口和触发条件；
3. 页面布局与区域关系；
4. 完整操作步骤；
5. 展示内容、字段和来源；
6. 产品规则与信息优先级；
7. 交互反馈；
8. 默认、加载、成功、失败、空、禁用、部分成功等状态；
9. 异常、恢复、重试、撤销、覆盖、追加和只读规则；
10. 结果与数据去向；
11. 跨页面、跨角色和跨端影响；
12. 验收标准。

复杂机制使用连续步骤和短段落。表格只用于字段、状态、对照、数据去向、指标和权限，不得用三四行摘要表代替完整产品行为。

页面布局可以使用 ASCII 线框、清晰布局图或原型截图。原型中的左右、上下、固定、折叠、悬浮、抽屉和返回关系均视为产品规则。

## 5. 推荐正文结构

1. 产品概述
2. 目标用户与使用场景
3. 产品结构与页面关系
4. 完整产品流程与核心用户流程
5. 产品内部运作逻辑
6. 通用组件与交互规则
7-N. 按用户任务顺序展开各业务模块
N+1. 跨功能业务规则、异常处理与关键文案
N+2. 数据流与系统集成
N+3. AI 功能与内部处理规则（如适用）
N+4. 质量监控与成功标准
N+5. 非功能需求
N+6. 待确认问题

不要机械保留空章节，但完整流程、内部运作和详细业务模块不可删除。

使用 `assets/templates/PRODUCT_FUNCTION_PRD.md` 作为起始骨架，不得照抄占位内容。

## 6. 工程噪音与产品规则分界

主 PRD 可以写状态如何迁移、数据从哪里来和确认后去哪里、异步任务对用户如何表现、幂等与并发对用户行为的约束、AI 输入优先级、人工确认和错误红线。

主 PRD 不应大面积写 API 路径、完整请求响应、数据库表、UUID、索引、DDL、消息队列实现、完整 Prompt、JSON Schema、模型路由、埋点事件列表、部署和基础设施细节。这些进入工程规格、AI 规格或 QA 文档，不得反向压缩 PRD 的功能细节。

## 7. 强制交付工作流

1. 读取权威资料并建立内部规则清单。
2. 建立“来源规则 → PRD 章节”内部覆盖表。
3. 先完成目录、完整产品总流程、核心用户流程和内部运作图。
4. 再逐模块写详细功能。
5. 生成最终 DOCX，不以中间 Markdown 作为正式交付物。
6. 运行：`python scripts/audit_prd.py <最终PRD.docx> --strict`。
7. 改写既有文档时运行：`python scripts/audit_prd_coverage.py <权威来源> <最终PRD.docx> --strict`。
8. 任一审查 FAIL 时修正并重新审查，禁止交付失败文档。
9. 按 DOCX 工具要求渲染全部页面，逐页检查流程图、表格、标题、页眉页脚、断行、遮挡和空白页。
10. 只有脚本 PASS 且视觉检查通过后才能交付。

## 8. 最终回复

只说明已生成的 PRD、权威来源覆盖情况、三类流程图和业务模块、审查与渲染结果、下载链接。不要再次写产品教练过程、版本修订史或阶段治理。
