# 产品功能验证与设计验证开发（完整保留版）

目标不是粗糙 Demo，也不是生产上线，而是在底层服务可模拟的前提下，把全部用户可见产品行为尽可能真实地实现，使产品经理能够通过操作发现设计问题并继续完成产品设计。

## 1. 开始前必须完成读取

1. 读取 `PRODUCT_TRUTH.md`、`DESIGN_PRINCIPLES.md`、`DECISIONS.md`、`OPEN_QUESTIONS.md` 和 `CURRENT_DELIVERABLES.md`。
2. 不得每次重新生成设计理念。只有出现新的长期判断、理念冲突、多轮稳定规律或用户明确要求时，才调用设计理念模块形成候选，并在用户确认后更新。
3. 读取唯一主 PRD、用户认可的原型和截图、已有工程与 AI 约束。
4. 读取已有模拟数据；缺失时依据 PRD 和工程规格生成覆盖正常、空、失败、边界和跨端状态的数据。
5. 读取当前项目的 package.json、源码、路由、状态管理、已有页面、构建和测试方式。

不得直接从当前代码反推产品目标。

## 2. 固定权威顺序

除用户本轮明确变更外：

`用户本轮明确决定 > 设计理念 > PRD > 工程与 AI 约束 > 模拟数据 > 当前项目文件 > 通用产品惯例`

遇到冲突时：
- 不静默折中；
- 记录到 `SOURCE_CONFLICTS.md`；
- 按更高优先级实现；
- PRD 与理念冲突时优先理念，并指出 PRD 需要修订；
- 当前代码与理念或 PRD 冲突时允许重构，但必须保留可恢复备份；
- 重大产品歧义登记到 `OPEN_QUESTIONS.md`，不得自行决定。

## 3. 默认交付形态

当用户要求完整开发包时，至少生成或更新：

1. `SOURCE_AUTHORITY.md`；
2. `SOURCE_CONFLICTS.md`；
3. 产品工程规格；
4. AI 功能规格（存在 AI 功能时）；
5. `FEATURE_IMPLEMENTATION_MATRIX.md`；
6. `MOCK_DATA_SPEC.md` 与可直接使用的 JSON、CSV 或媒体占位数据；
7. `CODEX_MASTER_TASK.md`；
8. `DEVELOPMENT_GUIDE.md`；
9. `QA_ACCEPTANCE.md`；
10. `CODEX_RESUME_PROMPT.md`；
11. 产品经理操作与功能验证指南；
12. `DECISIONS.md` 中的开发补充建议；
13. `OPEN_QUESTIONS.md` 中未授权的重大产品问题。

项目已有对应文件时更新，不重复制造多个冲突版本。

## 4. 全功能设计验证标准

- 产品行为真实，底层 AI、上传、转码、存储、通知和数据库可以模拟；
- 所有主功能至少达到 L3：可操作、有状态、有加载、空、失败、取消、重试、撤销和刷新恢复；
- 发布、分配、学习、测试、作业、评分、讨论、成长、奖励和管理动作等关键跨端关系达到 L4；
- 多端共享统一模拟业务内核，禁止每页独立静态数组；
- 使用项目已有状态方案；缺失时使用状态库、模拟 API、localStorage 或 IndexedDB；
- 提供预设场景、事件日志、一键重置和刷新恢复；
- 建立隐藏产品实验室，默认 `Shift + D`；
- 产品经理不需要预先写体验卡，Codex 应从 PRD 建立页面—功能—状态—异常—跨端影响矩阵。

同时读取 `design-validation-standard.md` 和 `mock-data-standard.md`。

## 5. 产品实验室最低能力

至少支持：
- 角色和用户切换；
- 业务对象切换；
- 正常、空、失败、超时、低置信度、未配置和部分成功；
- 小、中、大数据规模；
- 跨端同步延迟和失败；
- 关键结果预置；
- 事件日志；
- 一键重置；
- 可复现数据种子。

产品实验室不是面向正式用户的后台，不得污染正常产品体验。

## 6. Codex 可以长时间执行，但必须内部可控

允许一个总任务、一个主分支和长时间自主执行。Codex 内部分阶段完成：

1. 现状审计；
2. 权威来源和冲突矩阵；
3. 产品工程规格与 AI 规格；
4. 统一模拟业务内核；
5. 各端完整功能；
6. 跨端联动；
7. 产品实验室；
8. 视觉、响应式和可访问性精修；
9. 全量 QA；
10. 构建、截图、录屏和一次性交付。

每个阶段保存可恢复提交。一般微交互不要求用户逐项确认；只有无法通过理念、PRD 和历史决定判断的重大产品规则才暂停或登记待确认。

## 7. 未定义细节与决策管理

- 微交互采用成熟、简单、自然、可撤销的默认方式；
- 不得新增重大业务规则、角色、审批、收费或权限体系；
- 微交互补充写入 `DECISIONS.md` 的“开发补充 / 建议”区，写明问题、选择、理由、替代方案、影响页面和修改位置；
- 只有用户确认后，开发补充才能成为正式产品事实；
- 重大歧义写入 `OPEN_QUESTIONS.md`，不得静默决定。

不得另建 `PRODUCT_DECISIONS.md` 或 `OPEN_PRODUCT_DECISIONS.md`。

## 8. 禁止敷衍

- 禁止无响应按钮；
- 禁止 Toast 代替完整流程；
- 禁止直接跳过中间状态；
- 禁止静态卡片冒充复杂功能；
- 禁止只实现成功状态；
- 禁止多端数据割裂；
- 禁止 TODO、FIXME、Coming Soon 和空白占位；
- 禁止用“只是 Demo”省略保存、撤销、失败、重试、刷新恢复和结果去向；
- 禁止为了复用旧代码而违背设计理念；
- 禁止声称测试或构建通过却没有实际命令结果。

## 9. 正式开发任务与质量门

使用 `assets/templates/CODEX_MASTER_TASK.md`。填写完整后运行：

```bash
python scripts/audit_design_validation_task.py CODEX_MASTER_TASK.md --strict
```

审查失败不得交付。

同时检查：
- 功能矩阵是否覆盖所有页面、操作、状态、异常和跨端影响；
- 模拟数据是否覆盖正常、空、失败、边界、规模和一致性；
- 工程规格是否没有反向改变产品；
- AI 规格是否定义人工确认、失败降级和高影响红线；
- 当前项目是否真实执行安装、类型检查、测试和 `npm run build`；
- 是否提供桌面和移动端截图或录屏。

## 10. 最终指导

向用户说明：
- 应将哪些文件放入 Codex 工作区；
- 如何启动长任务；
- 哪些权限可以授予，哪些生产凭证不能提供；
- 中断后如何恢复；
- Codex 最终必须交付哪些证据；
- 产品实验室如何使用；
- 产品经理如何按角色和场景操作，并记录“看不懂、觉得烦、不相信”的位置；
- 哪些能力仍属于客户试点或生产上线差距。
